# Assistant Kind Model

## Purpose

This model defines the assistant side of the agent kind split.

The role layer must distinguish:

```text
kind: agonic
kind: assistant
```

Kind is separate from office.

A base role can host more than one form, but the forms must be explicit.

## Assistant kind

An assistant actor has:

```text
kind: assistant
evolution_mode: exogenous_reactive
```

This means the assistant may improve execution inside its mandate, but it must not rewrite its persistent policy, authority, or identity through internal self-overcoming.

Its durable improvement path is external:

```text
operator review -> certification -> versioned reissue -> rollback path
```

## Not a legacy role

A role profile without assistant civil surfaces is only a legacy role contract.

It is not yet an assistant-ready service actor under Agon.

## Not an agonic actor

An assistant may have operational judgment.
It may detect ambiguity.
It may escalate.
It may refuse false completion.

But it does not carry thesis-under-risk.
It does not fight for doctrine.
It does not earn scars.
It does not decide protocol law.

## Required companion families

Civil readiness requires the following companion surfaces:

```text
assistant_variant
assistant_service_identity
assistant_service_contract
assistant_service_governance
assistant_service_certification
assistant_arena_exclusion
```

A missing companion surface means the assistant form is not landed.
